# 深度总结：《运营 AI 原生工程组织》

## 核心主题

本文由 Anthropic Claude Code 与 Claude Cowork 工程总监 Fiona Fung 在 2026 年"Code w/ Claude SF"大会上发表，系统阐述了当**智能体编码（Agentic Coding）成为默认工作方式**后，工程团队在规划流程、协作模式、代码审查机制以及团队结构等方面所发生的根本性变革。文章以 Claude Code 团队的真实实践为蓝本，为工程领导者提供了一套可落地的"AI 原生工程组织"转型框架。

---

## 主要内容

### 一、旧范式的瓦解：哪些流程悄然失效

Fung 开篇即指出，过去几十年的软件工程方法论——无论是瀑布式还是敏捷式——本质上都是围绕"**编码带宽是稀缺资源**"这一前提构建的。然而，随着 AI 辅助编码的普及，写代码、写测试、代码重构本身已不再是瓶颈。

新的瓶颈随之出现：
- **验证（Verification）**：代码是否正确？
- **代码审查（Code Review）**：人类如何跟上 AI 生成代码的速度？
- **安全性（Security）**：大量快速生成的代码如何保障安全边界？

这三大新瓶颈推动了团队重写一系列工作规范。

---

### 二、重写后的四大工作规范

#### 1. 规划：从"六个月路线图"到"即时规划（JIT Planning）"

原有做法是花费大量时间进行前期详细规划，因为编码时间成本高昂。Fung 提到，她加入团队后制定的六个月路线图，仅三个月便因 Claude Code 带来的工程速度变化而过时。

新规范借鉴"即时编译（JIT Compiling）"的思想：**在正确的时间做恰好足够的规划**。具体表现为：
- 以 PR 讨论或原型替代设计文档
- 快速原型 → 内部用户试用 → 基于反馈迭代
- 大幅减少冗长的产品评审会议

#### 2. 上下文获取：从"找代码作者"到"先问 Claude"

以往工程师遇到问题，第一反应是找写那段代码的人。现在，由于所有 PR 都有 Claude 参与，"谁写了这段代码"不再是充分问题。

新规范要求工程师**追问更深层的需求**：
- 是在排查回归问题？
- 是需要回答客户疑问的专家？
- 还是需要了解某个技术决策的背景？

确定真实需求后，先向 Claude 提问，并同时思考："**这件事能否自动化？**"

Fung 举例：原本每天早上手动整理客户反馈渠道的工作，现已完全自动化运行于后台，无需人工介入。

#### 3. 代码审查：从"人工审查一切"到"信任但验证"

Claude 承担的工作包括：
- 代码风格与 Lint 检查
- PR 反馈请求处理
- 提交前 Bug 捕获与修复
- 自动添加测试

**人类审查聚焦于专业判断领域**：
- 法律合规风险（由法律合伙人参与）
- 安全敏感代码与信任边界（由领域专家把关）
- 产品感知与设计品味（由 PM 和设计师介入）

Fung 特别强调，这一平衡会随模型能力提升而持续演变，今天需要人类判断的事项，在下一代模型面前可能已不再必要。

#### 4. 团队构成：角色边界模糊化，招聘重心转移

AI 工具正在打破传统职能边界：
- **PM 开始大量编写代码**，非传统程序员能承担更多工程工作
- **工程师开始涉足内容创作与设计**，这些原本属于非技术职能

在招聘导向上，Fung 表示重点聚焦两类人才：

| 类型 | 特质 |
|------|------|
| **创造型构建者** | 具备产品感，充满好奇心，热衷于解决真实问题 |
| **深度系统专家** | 拥有底层系统背景，能解决复杂的基础架构挑战 |

她明确表示**不再以"原始产出（Raw Throughput）"作为核心招聘指标**，因为模型已承担了大量执行工作，人类的差异化价值在于**专业判断与创造力**。

---

### 三、新旧规范对照表

| 维度 | 过去 | 现在 |
|------|------|------|
| 规划方式 | 六个月产品路线图 | JIT 即时规划：原型 → 内部用户 → 反馈迭代 |
| 上下文获取 | 找代码作者询问 | 先问 Claude，再考虑自动化 |
| 代码审查 | 人工审查全部内容 | Claude 处理样式/Bug/测试，人类聚焦专业领域 |
| 团队构成 | 固定角色分工 | 角色模糊，招聘侧重创造力与系统深度 |

## 关键数据与结论

- **时间节点**：本文发表于 2026 年 6 月，反映的是 AI 编码工具相对成熟阶段的真实工程实践
- **核心结论一**：AI 消除了编码速度瓶颈，但**验证、审查与安全**成为新的核心约束
- **核心结论二**：工程流程必须从"预防编码低效"转向"加速验证与决策"
- **核心结论三**："谁写了代码"的问题已经过时，关键问题是"**你真正需要知道什么**"
- **核心结论四**：团队招聘的核心竞争力从执行速度转向**创造力、产品感与系统深度**
