# Agent Identity Access Model：Claude 的自主多人协作访问模型深度解析

## 核心主题

本文介绍了 Anthropic 为其多人协作 AI 产品 **Claude Tag** 设计的全新访问控制模型——**Agent Identity（智能体身份）**。该模型旨在解决当 AI 从"单人助手"演进为"团队协作成员"时，传统"以用户身份行事"权限机制所暴露的根本性缺陷，并提供一套更适配自主、多人场景的权限管理框架。

---

## 主要内容

### 一、问题背景：为何传统权限模型失效

在传统的单人 AI 使用场景中，用户将自己的账户（如 Google Drive、GitHub、日历）授权给 AI，AI 以该用户身份执行操作，逻辑清晰、边界明确。然而，当 AI 进入**多人共享工作空间**（如 Slack 频道）时，这一模型面临两大根本性挑战：

1. **智能体自主性的提升**：AI 完成任务的能力正以每四个月翻倍的速度增长。如今的智能体可以自主调度任务、响应事件，甚至在发起请求的用户下线之后继续独立工作。在这种异步、自主的工作模式下，"代表某一用户行事"的前提已不再成立。

2. **多人协作的权限困境**：当一个频道中有多名工程师、产品经理共同参与时，AI 究竟应该使用谁的权限？不存在任何一个"永远正确"的单一用户选择。沿用个人权限不仅逻辑混乱，还可能导致共享频道成为渗透他人私人文档的"后门"。

### 二、Agent Identity 模型的核心设计

**Agent Identity** 的核心理念是：**将权限的主体从"用户"转变为"智能体在特定空间中的身份"**，即以"这个智能体在这个空间里能做什么"替代"这个用户能做什么"。

具体机制如下：

- **Claude 拥有独立账户**：在 Claude Tag 激活的频道中，Claude 不代表任何用户行事，而是以自身身份存在——在 Slack 中以"Claude 应用"身份发帖，在 GitHub 中以"Claude GitHub App"身份发起 Pull Request，在数据仓库中使用管理员预配的服务账户进行查询。

- **工作区级别的基础身份继承**：管理员在工作区层面定义 Claude 的基础身份（即默认的连接和能力集合），所有频道默认继承这一身份。

- **频道级别的细粒度覆盖**：在此基础上，管理员可以按需为特定频道进行定制化配置，例如：为工程师频道授予 GitHub 和数据仓库访问权限，或将 CRM 连接限定在某一私密频道内。

管理员可配置的内容涵盖四个维度：
1. **仓库访问权限**：Claude 可读写哪些代码仓库
2. **连接器（Connectors）**：Claude 使用的工具和 API 密钥（同一服务可在不同频道配置不同权限级别，如数据仓库在普通频道只读、在数据团队私密频道可写）
3. **技能与插件（Skills & Plugins）**：Claude 动态加载的指令、脚本和资源，用于提升专项任务表现
4. **常驻指令（Standing Instructions）**：每个频道的自定义说明和背景上下文

### 三、身份边界的隔离机制

Agent Identity 模型在隔离设计上体现了清晰的安全边界逻辑：

- **私密频道拥有独立身份，公共频道共享工作区级身份**：法律频道的 Claude 身份无法触及未被授权的代码库，工程频道的 Claude 身份无法读取法律文档。
- **记忆隔离**：Claude 在私密频道中获取的信息不会出现在更广泛的工作区中，内存与访问权限均严格遵循频道边界。
- **企业级角色访问控制（RBAC）**：Enterprise 计划用户可进一步限定哪些成员有权调用 Claude，实现"能访问什么"与"谁能发问"的双重管控。
- **撤销便捷性**：由于 Claude 拥有独立身份，撤销该身份即可同步终止其在所有相关空间的访问权限，无需逐一审计分散在数十个用户账户中的操作记录。

### 四、广泛工具接入的价值乘数效应

Anthropic 在内部使用 Claude Tag 的实践表明，**工具和上下文的接入具有复利效应**：每接入一个新系统，都会让其他已接入系统变得更有价值，因为 Claude 能够跨系统整合信息——从 Slack 提取讨论线索、从 Drive 获取文档、从工单系统读取任务、从数据仓库执行查询，最终综合生成任何单一工具都无法提供的完整答案。

## 关键数据与结论

- **AI 任务能力的增长速度**：AI 智能体能够独立完成的任务时长，**每四个月翻倍一次**，这是 Anthropic 重新设计访问模型的核心驱动因素之一。
- **核心结论**：传统的用户权限委托模型（ACL）已不适用于自主、多人协作的 AI 场景；以"智能体身份"为核心的分隔式权限模型是当前最可行的替代方案。
- **安全边界设计**：频道成员即使本人没有直接访问某仓库的权限，也可以通过 Claude 读取该仓库——前提是该频道的配置文件授予了 Claude 相应权限。Anthropic 承认这是一个"非常规设计"，但认为它是实现自主多人智能体访问模型的必要一步。

---

## 实际意义

**对企业管理员而言**，Agent Identity 模型提供了一套更清晰、更易于治理的 AI 权限框架：无需为每个用户单独配置 AI 权限，而是通过统一的智能体身份管理实现集中控制，极大降低了权限蔓延和安全审计的复杂度。

**对团队协作而言**，该模型真正实现了 AI 作为"团队成员"而非"个人助手"的角色定位。Claude 在多人频道中不再是某个用户的代理，而是具备独立身份、遵守工作区规则的协作参与者，从根本上消除了
