# 深度总结：《驾驭 Claude 的智能》——构建应用的三大核心模式

---

## 一、核心主题

本文由 Anthropic 官方发布，聚焦于一个核心命题：**如何在 Claude 持续进化的背景下，构建出能够平衡智能水平、响应延迟与运营成本的 AI 应用程序**。文章提出了三个关键设计模式，指导开发者在构建"智能体套件"（agent harness）时，充分释放 Claude 的内在能力，而非用过时的假设约束它。

---

## 二、背景与问题提出

Anthropic 联合创始人 Chris Olah 指出，像 Claude 这样的生成式 AI 系统，与其说是"被构建"的，不如说是"被培育"的——研究人员设定条件引导其成长，但具体涌现出什么能力并不总是可预测的。

这一特性给应用开发带来了结构性挑战：**智能体套件往往基于"Claude 做不到某事"的假设而设计，但随着 Claude 能力的提升，这些假设会逐渐失效**。因此，开发者需要定期重新审视这些假设，而非一劳永逸地固化系统架构。

---

## 三、三大核心模式详解

### 模式一：充分利用 Claude 已有的知识（Use What It Already Knows）

**核心理念**：基于 Claude 已经熟悉的工具构建应用，而非为每个任务发明专用工具。

文章以具体数据为证：2024 年底，Claude 3.5 Sonnet 仅凭一个 **bash 工具**和一个**文本编辑器工具**，在 SWE-bench Verified 基准测试中达到了 **49% 的得分**，达到当时的技术最优水平。Claude Code 的底层设计也基于这两种工具。

值得注意的是，bash 并非专为智能体设计，但 Claude 天然擅长使用它，且随着版本迭代持续提升。更重要的是，Claude 能够将这些通用工具**组合编排**，衍生出更高级的能力，例如：
- **Agent Skills**（智能体技能）
- **程序化工具调用**（Programmatic Tool Calling）
- **记忆工具**（Memory Tool）

这三者本质上都是 bash 与文本编辑器工具的组合应用。

---

### 模式二：主动追问"我可以停止做什么？"（Ask What You Can Stop Doing）

这一模式的哲学在于：**将决策权从硬编码的套件转移给 Claude 本身**，让模型在力所能及之处自主决策。

#### 2.1 让 Claude 自主编排行动

传统假设认为，每次工具调用的结果都应流回 Claude 的上下文窗口以指导下一步行动。但这样做既慢又贵，且往往不必要。

更好的做法是赋予 Claude 一个**代码执行工具**（如 bash 或特定语言的 REPL），让它自己编写代码来表达工具调用及其之间的逻辑关系。这样，Claude 可以自主决定哪些结果需要传递、哪些需要过滤、哪些可以直接通过管道传给下一个调用——**只有最终代码执行结果才进入上下文窗口**，大幅降低 token 消耗。

关键数据支撑：在 **BrowseComp** 基准测试（测试智能体网页浏览能力）中，赋予 Opus 4.6 自主过滤工具输出的能力后，准确率从 **45.3% 提升至 61.6%**，提升幅度高达 16.3 个百分点。

这一发现还引出一个重要推论：**强大的编码模型天然是强大的通用智能体**，因为代码是 Claude 编排行动最通用的方式。

#### 2.2 让 Claude 自主管理上下文

传统假设认为，系统提示词应由人工精心设计，预先加载大量任务指令。但这种方式难以扩展——每增加一个 token 都在消耗 Claude 的注意力预算，而大量预加载的指令往往很少被用到。

解决方案是引入 **Skills（技能）机制**：每个技能文件的 YAML 元数据（简短描述）预加载到上下文窗口，Claude 在需要时通过调用文件读取工具来**渐进式展开**完整技能内容，实现按需加载。

同时，**上下文编辑（Context Editing）** 提供了反向操作的能力——选择性移除已过时或不再相关的上下文（如旧工具结果或思考块）。**子智能体（Subagents）** 机制则允许 Claude 在处理特定任务时分叉出全新的上下文窗口，实现工作隔离。数据显示，在 BrowseComp 中，Opus 4.6 使用子智能体后，比最佳单智能体运行结果提升了 **2.8%**。

#### 2.3 让 Claude 自主持久化上下文

长期运行的智能体任务可能超出单个上下文窗口的限制。传统假设认为，记忆系统应依赖模型外部的检索基础设施。Anthropic 的研究方向则相反：**给 Claude 简单的工具，让它自己决定持久化什么内容**。

**压缩机制（Compaction）** 允许 Claude 对过去的上下文进行摘要，维持长任务的连续性。数据表明，不同模型版本的能力差异显著：在 BrowseComp 中，Sonnet 4.5 无论给予多少压缩预算，准确率始终维持在 **43%**；而 Opus 4.5 在相同配置下可扩展至 **68%**，Opus 4.6 更是达到了 **84%**。

**记忆文件夹（Memory Folder）** 是另一种方式，允许 Claude 将上下文写入文件并在需要时读取。在 **BrowseComp-Plus** 测试中，为 Sonnet 4.5 引入记忆文件夹后，准确率从 **60.4% 提升至 67.2%**，提升约 6.8 个百分点。
