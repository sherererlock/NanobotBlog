# CodeRabbit 如何利用 Claude 构建 Agent 编排系统

## 核心主题

本文介绍了 AI 代码审查平台 CodeRabbit 如何借助 Anthropic 的 Claude 模型，构建了一套在代码生成**之前**运行的智能编排与规划系统，以解决 AI 编码工具普遍存在的"代码能跑、但跑偏了"问题。

---

## 主要内容

### 一、问题起点：AI 代码的隐性失败模式

随着 AI 编码工具的普及，代码从想法到原型的速度大幅压缩，但 CodeRabbit 在分析其客户提交的大量 Pull Request 时，发现了一个反直觉的规律：**大量 AI 生成的代码能够编译通过、测试通过，却根本没有解决开发者真正想解决的问题。**

CodeRabbit VP of AI David Loker 指出，这一问题的根源并不在于模型本身，而在于**人类开发者与 AI 系统之间的"隐性知识鸿沟"**。

资深开发者长年积累的经验已深度内化，许多"显而易见"的前提条件从不会被写进需求文档。然而，当他们与 AI 编码代理协作时，却在无意识地对 AI 做出同样的经验预设——而 AI 没有这些上下文，只能用自认为"合理"的方式填补空白。

Loker 分享了一个亲身案例：他在个人项目中构建了一套记忆系统，与 AI 代理反复迭代数小时后系统终于运行，但当他询问使用方式时，AI 的说明要求传入用户 Token——而整个系统压根没有登录页面。原因很简单：他说过系统需要用户，但从未说明用户需要登录入口。AI 填补了这个空白，数小时的工作成果却缺少了最基础的"大门"。正如 Loker 所言：**"在 AI 工作流中，滞后的验证代价极为高昂。"**

---

### 二、解决方案：在代码生成前插入规划层

CodeRabbit 的应对策略是在代码生成环节之前，插入一个由多个 Claude 模型协同驱动的**Agent 编排与结构化规划系统**。

该系统的核心逻辑是：**规划质量决定输出质量**。代码生成越便宜，走错方向的代价就越昂贵——因此，在生成代码之前确保方向正确，是整个流程中最高价值的环节。

规划系统的最终产出是一份**协作式产品需求文档（PRD）**：

- 包含完整的上下文信息
- 经过团队各相关方的验证与审阅
- 在实际编码开始之前即已确定
- 明确记录"做了什么决定"以及"为什么这么决定"

Claude Code 随后接收该 PRD，并基于此生成精细化的实施计划。这份文档不仅帮助团队避免返工，还能在事后验证输出是否符合原始意图，并在新工程师入职时发挥知识传承的作用。

值得注意的是，Loker 特别强调：**"这套规划系统并非要替代 Claude Code 的 Plan Mode，而是更高层次的编排，发生在 Claude Code 介入之前，将方向限定在足够精确和正确的范围内。"**

---

### 三、跨模型路由：按任务复杂度匹配模型层级

CodeRabbit 在系统设计中对 Claude 模型家族进行了精细的分工路由，以在成本、延迟与质量之间实现最优平衡：

| 模型 | 角色 | 任务类型 |
|------|------|----------|
| **Claude Opus** | 编排核心 | 理解问题全貌、设定整体方向、驱动高层战略规划 |
| **Claude Sonnet** | 结构化规划 | 将 Opus 的输出序列化为具体的规划步骤 |
| **Claude Haiku** | 轻量执行 | 上下文蒸馏、特定工具调用等范围明确的窄域操作 |

团队的模型选择策略并非凭直觉，而是**基于评估数据驱动**的。Loker 表示："如果 Haiku 在某个任务上与 Sonnet 表现相当，我们就用 Haiku；如果评估框架显示给 Opus 更多空间能提升规划质量，我们就给它更多空间。我们不靠猜。"

---

### 四、构建规划质量评估体系

CodeRabbit 已拥有成熟的代码审查评估系统，但针对**规划输出的评估**却是全新挑战，需要从零构建。

评估体系的建立经历了几个阶段：

1. **人工标注起步**：从手工调优示例和人工检查开始，积累高质量样本
2. **LLM 裁判体系**：开发了一套 LLM 评判器库，从多个维度对规划质量进行打分
3. **端到端验证**：由于规划最终会生成代码，团队可以测量生成代码是否有效、是否超出预定范围、消耗了多少 Token
4. **对照实验**：对同一任务分别在有/无规划步骤的条件下运行，从而独立衡量规划本身带来的价值增益

评估过程中，团队还发现了规划粒度的"金发地带"难题：**规划过于细粒度**，代码库一旦变动就会过时；**规划过于高层**，又给 AI 留下太多自行填充的空间。找到恰当的细节层次，本身就是一项需要数据支撑的系统性工程。

---

## 关键数据与结论

- **规模**：CodeRabbit 每周处理超过 **200 万个 Pull Request**，服务 **15,000+ 客户**
- **成立时间**：2023 年，发展迅速
- **核心洞察**：AI 代码的主要失败模式不是技术错误，而是**方向性错误**——代码正确执行了错误的需求
- **核心原则**：在 AI 工作流中，**验证越晚，代价越高**；规划质量是决定最终输出质量的关键杠杆
- **模型策略**：通过数据驱动的评估框架动态路由三个模型层级，而非依赖经验判断
